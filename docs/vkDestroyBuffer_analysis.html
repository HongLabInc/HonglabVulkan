1. 검증 오류 (Validation Error)

[ERROR] VUID-vkDestroyBuffer-buffer-00922
vkDestroyBuffer(): can't be called on VkBuffer 0x4e000000004e
that is currently in use by VkCommandBuffer 0x1879aa85900

이 오류는 Ex11_PostProcessing의 초기화 단계에서, sky 파이프라인 생성 직후 staging 버퍼를 통해 텍스처를 로드하는 시점에 발생합니다.

2. 스펙 규정 vs 검증 레이어의 동작

Vulkan 스펙 (합법)

VUID-00922가 요구하는 것은 명령들이 실행 완료(completed execution) 상태여야 한다는 것입니다. vkWaitForFences가 반환된 시점에는 명령이 이미 완료된 상태이므로, 버퍼를 파괴하는 것은 스펙상 합법입니다.

스펙은 참조된 리소스를 파괴하기 전에 커맨드 버퍼를 먼저 free 해야 한다고 요구하지 않습니다. 단지 제출된 작업이 끝나기만 하면 됩니다.

검증 레이어 (더 엄격)

Khronos 검증 레이어는 "사용 중(in use)" 여부를 펜스 완료가 아닌 커맨드 버퍼의 생존 기간을 기준으로 추적합니다. 버퍼를 참조하는 커맨드 버퍼가 여전히 할당된 상태(free 또는 reset되지 않음)라면, 일부 레이어 버전은 그 버퍼를 "사용 중"으로 표시합니다.

이는 보수적인 휴리스틱으로, 펜스가 시그널된 상태에서도 커맨드 버퍼가 살아있다면 거짓 양성(false positive)을 발생시킬 수 있습니다.

2b. Vulkan 스펙 심층 분석

이 상황에는 세 가지 별개의 스펙 조항이 적용됩니다. 이 셋을 종합하면 vkWaitForFences 이후에 버퍼를 파괴하는 것이 — 커맨드 버퍼가 여전히 할당되어 있더라도 — 합법임을 증명할 수 있습니다.

규칙 1 — VUID-vkDestroyBuffer-buffer-00922

vkDestroyBuffer(3) — Vulkan Spec

"buffer를 직접 또는 VkBufferView를 통해 참조하는 모든 제출된 명령은 반드시 실행이 완료되어 있어야 한다."

요구사항은 "실행 완료(completed execution)"이지 "커맨드 버퍼 free" 또는 "커맨드 버퍼 reset"이 아닙니다. 이것은 GPU 작업에 관한 규정이지, CPU 측 객체의 수명에 관한 규정이 아닙니다.

규칙 2 — 펜스 완료 보장

Vulkan 1.4 Spec — Synchronization

"펜스나 세마포어가 시그널되기 전에, 이전에 제출된 모든 큐 작업이 실행 완료되었음이 보장되며, 해당 큐 작업의 메모리 쓰기는 이후 큐 작업에서 가시(available)해진다."

vkWaitForFences가 VK_SUCCESS를 반환한 후에는, 해당 제출의 모든 명령이 실행 완료되었음을 스펙이 보장합니다. 이는 규칙 1을 직접적으로 충족시킵니다.

규칙 3 — 커맨드 버퍼 상태 전이

Vulkan Spec — Command Buffer Lifecycle

"커맨드 버퍼의 모든 제출이 실행 완료되면, 커맨드 버퍼는 pending 상태에서 다시 executable 상태로 돌아간다."

펜스가 시그널된 후, 커맨드 버퍼는 더 이상 pending 상태가 아니라 executable 상태로 돌아갑니다. 스펙의 객체 수명(Object Lifetime) 절은 다음과 같이 명시합니다:

"다음 Vulkan 객체들은 그 객체를 사용하는 어떤 커맨드 버퍼라도 pending 상태에 있는 동안에는 반드시 파괴되어서는 안 된다."

제약은 오직 pending 상태에 한정됩니다. 여전히 버퍼에 대한 내부 참조를 가지고 있는 executable 상태의 커맨드 버퍼는 — 스펙상 — 그 버퍼의 파괴를 막지 못합니다.

간극: 스펙 vs 검증 레이어

질문 Vulkan 스펙 검증 레이어
vkWaitForFences 후에 버퍼를 파괴해도 되는가? 가능 — 명령이 완료됨 오류로 표시할 수 있음
cmd buffer를 먼저 free 해야 하는가? 아니오 — 요구되지 않음 권장 — 거짓 양성 방지
"사용 중"은 어떻게 추적되는가? pending 상태 (GPU 실행 중) cmd buffer 할당 여부 (보수적)
무엇이 검사를 통과시키는가? fence 시그널 → 작업 완료 fence 시그널 + cmd free/reset

검증 레이어가 보수적 휴리스틱을 사용하는 이유는, 여러 큐 패밀리에 걸쳐 펜스 완료와 개별 리소스를 정확히 연관시키는 것이 복잡하기 때문입니다. 커맨드 버퍼 수명을 기준으로 추적하는 편이 더 단순하고 (실제 버그를 잡는 데에도) 더 안전합니다. 그 대가로 스펙상으로는 정확한 코드에서도 가끔 거짓 양성이 발생할 수 있습니다.

3. 업계 Best Practice

대표적인 Vulkan 예제 코드베이스 두 곳 모두 동일한 패턴을 사용합니다: flush 헬퍼 안에서 커맨드 버퍼를 free 한 뒤, 호출 측에서 staging 리소스를 파괴한다.

Sascha Willems / Vulkan 13k+ stars

base/VulkanDevice.cpp · flushCommandBuffer() 
void VulkanDevice::flushCommandBuffer(
    VkCommandBuffer commandBuffer,
    VkQueue queue, VkCommandPool pool,
    bool free /* = true */)
{
    vkEndCommandBuffer(commandBuffer);

    VkFence fence;
    vkCreateFence(device, &fenceInfo, nullptr, &fence);
    vkQueueSubmit(queue, 1, &submitInfo, fence);
    vkWaitForFences(device, 1, &fence, VK_TRUE, timeout);
    vkDestroyFence(device, fence, nullptr);
    if (free) {
        vkFreeCommandBuffers(device, pool, 1,
                             &commandBuffer);
    }
}

핵심 패턴

두 코드베이스 모두 깔끔한 관심사의 분리(separation of concerns)를 달성합니다: flush 헬퍼는 커맨드 버퍼의 수명을 책임지고, 호출 측은 staging 버퍼의 수명을 책임집니다. 커맨드 버퍼는 항상 staging 리소스가 파괴되기 전에 free 됩니다.

4. 현재 코드 — submitAndWait()

engine/CommandBuffer.cpp 
void CommandBuffer::submitAndWait()
{
    vkEndCommandBuffer(handle_);

    VkFence fence;
    vkCreateFence(device_, &fenceInfo, nullptr, &fence);
    vkQueueSubmit2(queue_, 1, &submitInfo, fence);
    vkWaitForFences(device_, 1, &fence, VK_TRUE, 1e9);
    vkDestroyFence(device_, fence, nullptr);

    // ← 이 시점에 커맨드 버퍼가 free 되지 않음
    // ← 나중에 ~CommandBuffer() → cleanup()에서 free 됨
}

Best practice 패턴과 달리, submitAndWait()가 반환된 후에도 커맨드 버퍼는 여전히 할당된 상태로 남습니다. 리소스 파괴 순서는 전적으로 C++ RAII 소멸 순서에 의존하게 됩니다.

5. C++ 소멸 순서 — 왜 중요한가

C++은 지역 변수를 선언의 역순으로 파괴합니다. 마지막에 선언된 변수가 가장 먼저 파괴됩니다.

"cmd가 살아있다는 사실"이 검증 레이어에게 왜 중요한가

vkWaitForFences 이후 GPU 작업은 완료된 상태입니다. 버퍼의 데이터는 더 이상 읽히지 않습니다. 스펙상 파괴는 합법입니다.

그러나 커맨드 버퍼 객체는 여전히 할당된 상태이고 Executable 상태에 있습니다. 내부적으로 버퍼에 대한 참조를 보유하고 있죠. 검증 레이어의 추적기는 이렇게 봅니다: "이 VkBuffer는 할당된 VkCommandBuffer가 참조하고 있다 → 사용 중이다." 펜스 대기와 리소스별 완료를 연관시키지 않습니다.

이것이 바로 보수적 거짓 양성입니다 — 여러 큐에 걸쳐 펜스와 리소스 완료를 정확히 추적하는 것이 복잡하기 때문에 레이어가 안전한 쪽을 택한 결과입니다.

6. 왜 예전에는 동작했나

요인 설명
Vulkan SDK 버전 검증 레이어는 SDK가 갱신될 때마다 함께 업데이트됩니다. 비교적 새로운 버전(~1.3.236+)은 리소스 수명 추적을 개선했지만 동시에 더 엄격한 검사를 도입했습니다. 최근 SDK 업데이트가 이 VUID 검사를 더 적극적으로 활성화했을 수 있습니다.
NVIDIA 드라이버 드라이버 업데이트는 내부적인 핸들 할당 방식과 타이밍을 바꿉니다. 다른 드라이버 버전에서는 커맨드 버퍼 핸들이 레이어에 의해 다르게 추적될 수 있습니다.
분리된 전송 큐 RTX 4090은 전용 전송 큐(패밀리 1)를 가지고 있습니다. 이전 GPU들이 그래픽스와 전송에 같은 큐 패밀리를 사용했다면, 레이어의 펜스 추적이 더 단순했고 오발(misfire) 가능성도 낮았을 것입니다.
비결정적 동작 이 오류는 레이어 내부 부기(bookkeeping) 상태에 의존합니다. 핸들 재활용, 할당 순서, 타이밍 등이 모두 레이어가 특정 실행에서 이 문제를 표시할지 여부에 영향을 줍니다.
자주 테스트되지 않음 이 문제는 항상 잠재되어 있었지만, 이 특정 예제를 검증을 켠 채로 자주 실행하지 않았기 때문에 관찰되지 않았을 수 있습니다.

7. 해결 방법

engine/CommandBuffer.cpp — submitAndWait() 
void CommandBuffer::submitAndWait()
{
    vkEndCommandBuffer(handle_);

    VkFence fence;
    vkCreateFence(device_, &fenceInfo, nullptr, &fence);
    vkQueueSubmit2(queue_, 1, &submitInfo, fence);
    vkWaitForFences(device_, 1, &fence, VK_TRUE, 1e9);
    vkDestroyFence(device_, fence, nullptr);

    vkFreeCommandBuffers(device_, commandPool_,
                         1, &handle_);     // ← 추가
    handle_ = VK_NULL_HANDLE;              // ← ~CommandBuffer에서 이중 free 방지
}

이 수정이 달성하는 것

  • Sascha/Khronos 패턴과 정확히 일치 — flush 함수 안에서 cmd가 free 됨
  • 모든 경로를 한 번에 해결 — 호출 측이 RAII를 쓰든 명시적으로 vkDestroyBuffer를 호출하든, cmd는 이미 free된 상태
  • 검증 거짓 양성 제거 — 리소스 참조를 보유한 할당된 cmd가 존재하지 않음
  • 기존 RAII와도 안전 — handle_ = VK_NULL_HANDLE 덕분에 소멸자의 cleanup()이 no-op으로 동작
  • 의미가 명확 — submitAndWait는 "fire and forget"을 뜻합니다. 호출 후 cmd는 재사용되지 않으므로 free 하는 것이 올바른 수명 관리입니다
