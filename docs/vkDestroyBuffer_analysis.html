1. The Validation Error

[ERROR] VUID-vkDestroyBuffer-buffer-00922
vkDestroyBuffer(): can't be called on VkBuffer 0x4e000000004e
that is currently in use by VkCommandBuffer 0x1879aa85900

This fires during initialization of Ex11_PostProcessing, right after sky pipeline creation, when loading textures via staging buffers.

2. What the Spec Says vs What the Layer Does

Vulkan Spec (Legal)

VUID-00922 requires commands to have completed execution. After vkWaitForFences returns, the commands have completed. Destroying the buffer is spec-legal.

The spec does not require the command buffer to be freed before destroying referenced resources — only that submitted work has finished.

Validation Layer (Stricter)

The Khronos validation layer tracks "in use" by command buffer lifetime, not solely by fence completion. If a command buffer that references a buffer is still allocated (not freed/reset), some layer versions flag the buffer as "in use."

This is a conservative heuristic — it can produce false positives when the fence has signaled but the command buffer is still alive.

2b. Vulkan Spec Deep Dive

Three separate pieces of spec language govern this situation. Together they prove that destroying a buffer after vkWaitForFences is legal — even if the command buffer is still allocated.

Rule 1 — VUID-vkDestroyBuffer-buffer-00922

vkDestroyBuffer(3) — Vulkan Spec

"All submitted commands that refer to buffer, either directly or via a VkBufferView, must have completed execution."

The requirement is "completed execution" — not "command buffer freed" or "command buffer reset." This is a statement about GPU work, not about CPU-side object lifetime.

Rule 2 — Fence Completion Guarantee

Vulkan 1.4 Spec — Synchronization

"Before a fence or semaphore is signaled, it is guaranteed that any previously submitted queue operations have completed execution, and that memory writes from those queue operations are available to future queue operations."

After vkWaitForFences returns VK_SUCCESS, the spec guarantees that all commands in that submission have completed execution. This directly satisfies Rule 1.

Rule 3 — Command Buffer State Transition

Vulkan Spec — Command Buffer Lifecycle

"Once execution of all submissions of a command buffer complete, it moves from the pending state, back to the executable state."

After the fence signals, the command buffer is no longer in pending state — it's back in executable state. The spec's Object Lifetime section says:

"The following Vulkan objects must not be destroyed while any command buffers using the object are in the pending state."

The restriction is on pending state only. An executable command buffer that still holds internal references to a buffer does not prevent that buffer from being destroyed — per the spec.

The Gap: Spec vs. Validation Layer

Question Vulkan Spec Validation Layer
Can I destroy a buffer after vkWaitForFences? Yes — commands completed May flag as error
Must the cmd buffer be freed first? No — not required Recommended — avoids false positive
How is "in use" tracked? By pending state (GPU executing) By cmd buffer allocation (conservative)
What satisfies the check? Fence signals → work done Fence signals + cmd freed/reset

The validation layer uses a conservative heuristic because accurately correlating fence completion with every individual resource across multiple queue families is complex. It's simpler (and safer for catching real bugs) to track by command buffer lifetime. The trade-off is occasional false positives for code that is spec-correct.

3. Industry Best Practice

Both major Vulkan example codebases use the same pattern: free the command buffer inside the flush helper, then destroy staging resources at the call site.

Sascha Willems / Vulkan 13k+ stars

base/VulkanDevice.cpp · flushCommandBuffer() 
void VulkanDevice::flushCommandBuffer(
    VkCommandBuffer commandBuffer,
    VkQueue queue, VkCommandPool pool,
    bool free /* = true */)
{
    vkEndCommandBuffer(commandBuffer);

    VkFence fence;
    vkCreateFence(device, &fenceInfo, nullptr, &fence);
    vkQueueSubmit(queue, 1, &submitInfo, fence);
    vkWaitForFences(device, 1, &fence, VK_TRUE, timeout);
    vkDestroyFence(device, fence, nullptr);
    if (free) {
        vkFreeCommandBuffers(device, pool, 1,
                             &commandBuffer);
    }
}

// Submit, wait, free cmd — all inside:

The Pattern

Both codebases achieve a clean separation of concerns: the flush helper owns the command buffer lifecycle, while the caller owns the staging buffer lifecycle. The command buffer is always freed before staging resources are destroyed.

void CommandBuffer::submitAndWait()
{
    vkEndCommandBuffer(handle_);

    VkFence fence;
    vkCreateFence(device_, &fenceInfo, nullptr, &fence);
    vkQueueSubmit2(queue_, 1, &submitInfo, fence);
    vkWaitForFences(device_, 1, &fence, VK_TRUE, 1e9);
    vkDestroyFence(device_, fence, nullptr);

    // ← command buffer NOT freed here
    // ← freed later by ~CommandBuffer() → cleanup()
}

Unlike the best-practice pattern, the command buffer remains allocated after submitAndWait() returns. Resource destruction depends entirely on C++ RAII destruction order.

5. C++ Destruction Order — Why It Matters

C++ destroys local variables in reverse order of declaration. The variable declared last is destroyed first.

Why "cmd still alive" matters to the validation layer

After vkWaitForFences, the GPU work is finished. The buffer data is no longer being read. Per the spec, destroying it is legal.

However, the command buffer object is still allocated and in Executable state. It holds internal references to the buffer. The validation layer's tracking sees: "this VkBuffer is referenced by an allocated VkCommandBuffer → it's in use." It does not correlate the fence wait with per-resource completion.

This is a conservative false positive — the layer errs on the side of caution because accurately tracking fence-to-resource completion across multiple queues is complex.

6. Why It Worked Before

Factor Explanation
Vulkan SDK Version Validation layers are updated with each SDK. Newer versions (~1.3.236+) improved resource lifetime tracking but also introduced stricter checks. A recent SDK update may have enabled this VUID check more aggressively.
NVIDIA Driver Driver updates change internal handle allocation and timing. A different driver version may cause the command buffer handle to be tracked differently by the layer.
Separate Transfer Queue RTX 4090 has a dedicated transfer queue (family 1). If previous GPUs used the same queue family for graphics and transfer, the layer's fence tracking was simpler and less likely to misfire.
Non-Deterministic The error depends on internal layer bookkeeping state. Handle recycling, allocation order, and timing can all affect whether the layer flags the issue on a given run.
Not Frequently Tested The issue may have always been latent but never observed because this specific example wasn't run often with validation enabled.

7. The Fix

engine/CommandBuffer.cpp — submitAndWait() 
void CommandBuffer::submitAndWait()
{
    vkEndCommandBuffer(handle_);

    VkFence fence;
    vkCreateFence(device_, &fenceInfo, nullptr, &fence);
    vkQueueSubmit2(queue_, 1, &submitInfo, fence);
    vkWaitForFences(device_, 1, &fence, VK_TRUE, 1e9);
    vkDestroyFence(device_, fence, nullptr);

    vkFreeCommandBuffers(device_, commandPool_,
                         1, &handle_);     // ← NEW
    handle_ = VK_NULL_HANDLE;              // ← prevents double-free in ~CommandBuffer
}

What This Achieves

  • Matches Sascha/Khronos pattern exactly — cmd freed inside the flush function
  • Fixes all paths — whether the caller uses RAII or explicit vkDestroyBuffer, the cmd is already freed
  • Eliminates validation false positive — no allocated cmd holds resource references
  • Safe with existing RAII — handle_ = VK_NULL_HANDLE makes the destructor's cleanup() a no-op
  • Semantically clear — submitAndWait means "fire and forget." The cmd is not reusable after this call, so freeing it is the correct lifecycle
