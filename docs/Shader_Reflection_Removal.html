런타임 Shader Reflection 제거

한 줄 요약: 앱 시작 시 SPIR-V를 파싱해서 바인딩 정보를 자동 추출하던 시스템을, C++ 코드에 직접 쓰는 명시적 방식으로 교체한다.
  • 기존 시스템은 프로덕션 수준으로 잘 만들어진 코드였다. 기술적 결함 때문이 아니라, 업계 흐름과 AI 도구의 발전 때문에 제거한다.
  • [주의] 강의 영상의 코드와 실제 코드가 다를 수 있습니다.

Shader Reflection이 뭔가요?

Vulkan에서 파이프라인을 만들려면 C++ 코드에서 "이 셰이더가 어떤 리소스를 사용하는지"를 정확히 기술해야 한다. 예를 들어 셰이더에 다음과 같은 코드가 있다면:

// 셰이더 (GLSL)
layout(set=0, binding=0) uniform SceneUBO { mat4 viewProj; };
layout(set=1, binding=0) uniform sampler2D albedoTex;
layout(set=1, binding=1) uniform sampler2D normalTex;

C++ 쪽에서도 동일한 정보를 기술해야 한다:

// C++ — 셰이더와 정확히 일치해야 함
VkDescriptorSetLayoutBinding bindings[] = {
    {0, VK_DESCRIPTOR_TYPE_UNIFORM_BUFFER,         1, VK_SHADER_STAGE_VERTEX_BIT},
    {0, VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER, 1, VK_SHADER_STAGE_FRAGMENT_BIT},
    {1, VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER, 1, VK_SHADER_STAGE_FRAGMENT_BIT},
};

Reflection이란, 컴파일된 SPIR-V 바이너리를 파싱하여 이 정보를 자동으로 추출하는 것이다. 셰이더만 작성하면 C++ 코드를 수정하지 않아도 파이프라인이 자동으로 구성된다.

기존 시스템이 한 일

ShaderManager가 spirv-reflect 라이브러리를 사용해서 앱 시작 시 자동으로 추출한 것:

  • 디스크립터 셋 레이아웃 — set/binding 번호, 타입, 셰이더 스테이지
  • 버텍스 인풋 포맷 — position, normal, UV 등의 메모리 레이아웃
  • 푸시 상수 범위 — 크기와 사용 스테이지
  • 컴퓨트 셰이더 워크그룹 크기 — dispatch 시 그리드 자동 계산
  • 읽기/쓰기 패턴 — 파이프라인 배리어 자동 삽입

이 구현이 프로덕션 수준인 이유

단순한 "바인딩 번호 읽기"를 넘어서, 실제 엔진에서 요구되는 고급 기능을 포함한다:

기능설명
레이아웃 중복 제거여러 파이프라인에서 같은 바인딩 구조를 쓰면 VkDescriptorSetLayout을 하나만 생성
스테이지 플래그 합산같은 바인딩이 vertex + fragment 셰이더 양쪽에서 쓰이면 stageFlags를 OR로 합침
쓰기 전용 감지SPIR-V 데코레이션을 검사해서 접근 패턴 판별
자동 배리어 정보BindingInfo에 targetLayout, targetAccess, targetStage를 저장해서 배리어 자동 삽입
가변 디스크립터 수배열 바인딩을 감지해서 PARTIALLY_BOUND 플래그 자동 설정

특히 자동 배리어 정보 추출은 프레임 그래프 기반 엔진에서나 볼 수 있는 수준이다. 분명히 해둘 것: 이것은 잘 만든 코드였고, 실제 문제를 해결했다.

왜 제거하는가

이유 1. AI가 Reflection의 역할을 대체한다

런타임 Reflection이 가치 있었던 이유는 사람의 실수 방지였다. 셰이더의 바인딩을 변경하면 C++ 쪽 코드가 자동으로 반영되므로, 양쪽을 수동으로 동기화할 필요가 없었다.

그러나 AI를 사용하는 환경에서는 이 장점이 약해진다:

  • AI가 셰이더를 읽고 대응하는 C++ 바인딩 코드를 생성할 수 있다
  • 셰이더와 C++ 양쪽을 동시에 수정할 수 있다
  • 불일치가 있으면 코드 리뷰 시 감지할 수 있다
  • "단일 진실의 원천(single source of truth)"이라는 Reflection의 핵심 이점이 희석된다

빌드타임 Reflection 도구(sokol-shdc, Oryol 셰이더 파이프라인)도 동일한 문제를 오프라인 코드 생성으로 해결한다. 그런데 AI가 셰이더와 C++ 코드를 동시에 생성한다면, 빌드 도구 자체도 불필요해진다.

이유 2. 업계가 런타임 Reflection에서 떠났다

실제 출시되는 게임 엔진들은 대부분 명시적 정의 또는 빌드타임 접근법을 쓴다:

엔진접근법
DOOM Eternal
(id Tech 7)		게임 전체에 약 12개 고정 디스크립터 레이아웃. 완전 bindless. 런타임 Reflection 없음.
출처: SIGGRAPH 2020 — "Rendering the Hellscape of DOOM Eternal"
The Forge
(ConfettiFX)		Shader Resource Table(SRT)로 런타임 Reflection 완전 제거. 셰이더와 C++가 같은 매크로 공유.
출처: FSL Programming Guide

Frostbite
(EA DICE)		프레임 그래프 + 빌드타임 메타데이터. 런타임 Reflection 없음.

(Google)		머티리얼 컴파일러(matc)가 빌드 시 바인딩 정보 삽입.

Reflection 라이브러리(SPIRV-Cross)의 저자조차 이렇게 말한다:

"게임 엔진 시나리오에서 SPIR-V를 런타임에 컴파일할 필요가 없다면, Reflection도 오프라인으로 옮기지 못할 이유가 없다."

"Bindless 디스크립터를 쓰거나 표준화된 파이프라인 레이아웃을 쓰면, Reflection은 사실상 의미가 없어진다."

— Hans-Kristian Arntzen, Granite Vulkan Backend Part 3

이유 3. 런타임 비용 대비 이득이 없다

본 프로젝트의 셰이더는 빌드 시점에 확정된다. 매 실행마다 동일한 SPIR-V를 파싱하여 동일한 결과를 얻는 것은 불필요한 비용이다:

비용내용
바이너리 크기spirv-reflect 라이브러리를 앱과 함께 배포해야 함
시작 시간매 실행마다 모든 셰이더의 SPIR-V를 파싱
디버깅 난이도"이 바인딩이 어디서 왔지?" → Reflection 코드를 추적해야 함
의존성외부 라이브러리를 빌드, 링크, 유지보수해야 함

이유 4. Reflection은 어려운 문제를 못 푼다

Reflection은 구조적 정보만 알려주고, 의미는 알려주지 못한다:

Reflection이 아는 것Reflection이 모르는 것
"binding 0은 uniform buffer, 128바이트""이건 카메라 행렬이다"
"binding 1은 combined_image_sampler""이건 쉐도우맵이다"
"set 0에 바인딩이 3개 있다""set 0은 프레임당 1번만 업데이트해야 한다"

결국 개발자는 의미를 직접 파악해야 한다. Reflection이 자동화하는 것은 기계적인 부분에 한정되며, 그 기계적인 부분은 명시적으로 작성하기 가장 쉬운 부분이기도 하다.

이유 5. 명시적 코드가 더 읽기 쉽다

Reflection 방식 — 바인딩의 출처를 추적해야 한다:

// 추적 경로: shader.spv → spirv-reflect → ShaderManager → DescriptorPool → Pipeline
auto pipeline = createPipelineFromReflection("pbrForward");

명시적 방식 — 코드 자체가 문서 역할을 한다:

VkDescriptorSetLayoutBinding bindings[] = {
    {0, VK_DESCRIPTOR_TYPE_UNIFORM_BUFFER,         1, VK_SHADER_STAGE_VERTEX_BIT | VK_SHADER_STAGE_FRAGMENT_BIT},
    {1, VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER, 1, VK_SHADER_STAGE_FRAGMENT_BIT},
    {2, VK_DESCRIPTOR_TYPE_STORAGE_BUFFER,         1, VK_SHADER_STAGE_FRAGMENT_BIT},
};

교육용 코드베이스에서 명시성은 부담이 아니라 장점이다.

이유 6. WebGPU의 설계가 이 방향을 뒷받침한다

WebGPU 스펙(2020–2024년, Vulkan의 교훈을 반영해서 설계)은 layout: 'auto'를 제공하지만, 프로덕션에서는 사용하지 말 것을 권장한다:

"자동 파이프라인 레이아웃은 바인드 그룹 공유를 막고, 셰이더를 수정하면 예측할 수 없는 방식으로 변할 수 있다. layout: 'auto'는 다른 파이프라인과 상태를 거의 공유하지 않는 일회성 파이프라인에만 쓸 것."

— Brandon Jones (Google Chrome WebGPU 팀), WebGPU Bind Group Best Practices
