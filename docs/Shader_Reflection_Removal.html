런타임 Shader Reflection 제거

한 줄 요약: 앱 시작 시 SPIR-V를 파싱해서 바인딩 정보를 자동 추출하던 시스템을, 파이프라인별 독립 클래스에 직접 정의하는 명시적 방식으로 교체했다. 
git checkout refactor/remove-shader-reflection
  • 기존 시스템은 프로덕션 수준으로 Reflection을 이용하여 자동화가 되어 있었다. AI 코딩 도구는 명시된 패턴을 보고 비슷하게 구현하는 것에는 매우 효율적이지만 자동화 알고리듬을 이해하는 데에는 상대적으로 비효율적이다. 특히 시스템 전체가 점점 더 복잡해질 경우 AI가 Reflection 관련 디버깅을 효율적으로 해내지 못할 가능성이 있다. 장기적으로 AI 도구 사용이 보편화될 것으로 예상되고 학생들이 공부하기에도 하나하나 명시하는 방식이 더 유리하기 때문에 Reflection을 삭제한다.

Shader Reflection이란

Vulkan에서 파이프라인을 만들려면 C++ 코드에서 "이 셰이더가 어떤 리소스를 사용하는지"를 정확히 기술해야 한다. 예를 들어 셰이더에 다음과 같은 코드가 있다면:

// 셰이더 (GLSL)
layout(set=0, binding=0) uniform SceneUBO { mat4 viewProj; };
layout(set=1, binding=0) uniform sampler2D albedoTex;
layout(set=1, binding=1) uniform sampler2D normalTex;

C++ 쪽에서도 동일한 정보를 기술해야 한다:

// C++ — 셰이더와 정확히 일치해야 함
VkDescriptorSetLayoutBinding bindings[] = {
    {0, VK_DESCRIPTOR_TYPE_UNIFORM_BUFFER,         1, VK_SHADER_STAGE_VERTEX_BIT},
    {0, VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER, 1, VK_SHADER_STAGE_FRAGMENT_BIT},
    {1, VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER, 1, VK_SHADER_STAGE_FRAGMENT_BIT},
};

Reflection이란, 컴파일된 SPIR-V 바이너리를 파싱하여 이 정보를 자동으로 추출하는 것이다. 셰이더만 작성하면 C++ 코드를 수정하지 않아�� 파이프라인��� 자동으로 구성된다.

기존 시스템이 한 일

ShaderManager가 spirv-reflect 라이브러리를 사용해서 앱 시작 시 자동으로 추출한 것:

  • 디스크립터 셋 레이아웃 — set/binding 번호, 타입, 셰이더 스테이지
  • 버텍스 인풋 포맷 — position, normal, UV 등의 메모리 레이아웃
  • 푸시 상수 범위 — 크기와 사용 스테이지
  • 컴퓨트 셰이더 워크그룹 크기 — dispatch 시 그리드 자동 계산
  • 읽기/쓰기 패턴 — 파이프라인 배리어 자동 삽입

프로덕션 수준이었던 이유

기능설명
레이아웃 중복 제거여러 파이프라인에서 같은 바인딩 구조를 쓰면 VkDescriptorSetLayout을 하나만 생성
스테이지 플래그 합산같은 바인딩이 vertex + fragment 셰이더 양쪽에서 쓰이면 stageFlags를 OR로 합침
쓰기 전용 감지SPIR-V 데코레이션을 검사해서 접근 패턴 판별

이것은 잘 만든 코드였고, 실제 문제를 해결했다.

제거 이유

1. AI가 Reflection의 역할을 대체한다

런타임 Reflection이 가치 있었던 이유는 사람의 실수 방지였다. 셰이더의 바인딩을 변경하면 C++ 쪽 코드가 자동으로 반영되므로, 양쪽을 수동으로 동기화할 필요가 없었다.

그러나 AI를 사용하는 환경에서는 이 장점이 약해진다:

  • AI가 셰이더를 읽고 대응하는 C++ 바인딩 코드를 생성할 수 있다
  • 셰이더와 C++ 양쪽을 동시에 수정할 수 있다
  • 불일치가 있으면 코드 리뷰 시 감지할 수 있다

5. 명시적 코드가 더 읽기 쉽다

Reflection 방식 — 바인딩의 출처를 추적해야 한다:

// shader.spv → spirv-reflect → ShaderManager → DescriptorPool → Pipeline
auto pipeline = createPipelineFromReflection("pbrForward");

명시적 방식 — 파이프라인 클래스를 열면 바로 보인다:

// ShadowMapPipeline.h — 이 파이프라인의 모든 바인딩 정보가 여기에
static auto layoutBindings() -> vector<vector<VkDescriptorSetLayoutBinding>> {
    return {{
        {0, VK_DESCRIPTOR_TYPE_UNIFORM_BUFFER, 1, VK_SHADER_STAGE_VERTEX_BIT, nullptr},
        {1, VK_DESCRIPTOR_TYPE_UNIFORM_BUFFER, 1, VK_SHADER_STAGE_VERTEX_BIT, nullptr},
        {2, VK_DESCRIPTOR_TYPE_UNIFORM_BUFFER, 1, VK_SHADER_STAGE_VERTEX_BIT, nullptr},
    }};
}

파이프라인별 독립 클래스

각 파이프라인이 자신의 바인딩 정보를 소유한다. 셰이더를 수정하면 해당 파이프라인 파일 하나만 업데이트하면 된다.

BindingInfo 헬퍼

BindingInfo.h에 정의된 헬퍼 함수로 배리어에 필요한 정보를 간결하게 생성한다:

bi::buffer("sceneData", 0, 0, VK_PIPELINE_STAGE_2_VERTEX_SHADER_BIT)
bi::sampledImage("shadowMap", 0, 10, VK_PIPELINE_STAGE_2_COMPUTE_SHADER_BIT)
bi::storageImageRead("floatColor1", 0, 3, VK_PIPELINE_STAGE_2_COMPUTE_SHADER_BIT)
bi::storageImageWrite("floatColor2", 0, 4, VK_PIPELINE_STAGE_2_COMPUTE_SHADER_BIT)

레이아웃 공유

ShaderManager::buildLayoutInfos()가 모든 파이프라인의 ��이아웃 바인딩을 모아서, 구조가 동일한 레이아웃은 하나의 VkDescriptorSetLayout으로 공유한다. 예: shadowMap set 0과 pbrDeferred set 0은 UBO 3개로 구조가 같으므로 하나의 레이아웃을 공유하고, stageFlags만 per-binding으로 OR한다.

Shader 클래스 변경

SPIR-V reflection 모듈을 제거하고, 셰이더 스테이지를 파일명에서 추론한다:

// .vert.spv → VK_SHADER_STAGE_VERTEX_BIT
// .frag.spv → VK_SHADER_STAGE_FRAGMENT_BIT
// .comp.spv → VK_SHADER_STAGE_COMPUTE_BIT
VkShaderStageFlagBits stage = Shader::stageFromFilename(spvFilename);

엔트리 포인트는 모든 셰이더가 "main"을 사용하므로 하드코딩한다.
